# NetaChoice アーキテクチャ

## 3ツール構成の狙い

NetaChoice は「ネタを集める → 仕分ける → 形にする」を、それぞれ独立した3つのツールで担う構成にする。
1つの大きなアプリにしないのは、各工程の責務を物理的に分離して、片方を壊さずもう片方を改修できるようにするため。

```
┌────────────────────────┐    ┌────────────────────────┐    ┌────────────────────────┐
│ neta-dataset-builder   │ →  │ neta-furui-lite        │ →  │ quote-video-factory    │
│ (収集・正規化)         │    │ (人手仕分けPWA)        │    │ (動画化)               │
└────────────────────────┘    └────────────────────────┘    └────────────────────────┘
            │                            │                              │
            └─────── shared/neta-schema.json (JSON交換フォーマット) ───┘
```

## 各ツールの責務

### neta-dataset-builder（新規・予定）

- **役割**: AI と Web 検索を使ってネタの候補を集め、`neta-schema.json` 準拠の JSON にして書き出す。
- **入力**: テーマ、収集元の指定、フィールド定義のテンプレート。
- **出力**: `shared/sample-dataset.json` と同じ形の JSON。
- **持たない機能**: 仕分け UI、動画生成、永続的なユーザーデータ管理。
- **想定実装**: Node CLI もしくは小さめのバックエンド。Claude API を呼ぶ。`status` フィールドは出力しない（仕分け前のため）。

### neta-furui-lite（既存・維持）

- **役割**: ユーザーが Tinder 風 UI でカードを 1 枚ずつ「残す / 捨てる / 保留」に仕分ける PWA。
- **入力**: `neta-schema.json` 準拠の JSON（インポート画面）。
- **出力**: 仕分け済み JSON（status 付き）、または「残す」だけを抽出した Markdown。
- **持たない機能**: AI 収集、Web スクレイピング、動画生成。これらは **本ツールに入れない**。
- **理由**: 仕分けは集中力のいる作業で、UI を軽く保ちたい。AI 機能を抱えると依存・ビルドサイズ・障害点が増える。

### quote-video-factory（新規・予定）

- **役割**: `status === "keep"` のカードを入力に、短尺動画（Shorts / Reels 想定）を生成する。
- **入力**: 仕分け済み JSON（neta-furui-lite からのエクスポート）。
- **出力**: 動画ファイル、字幕、サムネイル。
- **持たない機能**: ネタ収集、仕分け UI。
- **想定実装**: ffmpeg + 画像合成 + 音声合成。`provenance` フィールドを動画上の出典クレジットに反映する。

## 共有レイヤ

### shared/neta-schema.json

3ツール共通の JSON スキーマ（draft-2020-12）。破壊的変更は `schemaVersion` のメジャー値を上げて識別する。
各ツールはこのスキーマだけを信じればよく、互いの実装詳細を知る必要はない。

### shared/sample-dataset.json

スキーマに準拠した最小サンプル。3ツールの動作確認、CI のフィクスチャ、ドキュメントの実例として使う。
題材は「衛生講話ネタ」と「名言・引用集」の2データセット。

## 分割の境界線（迷ったときの判断基準）

| 機能 | どのツールに入れるか |
|---|---|
| Web からのネタ取得 | dataset-builder |
| Claude API 呼び出し | dataset-builder |
| 仕分けの状態管理 | furui-lite |
| ユーザー定義のカスタムフィールド編集 UI | furui-lite |
| ffmpeg / 画像合成 | video-factory |
| TTS / 字幕生成 | video-factory |
| スキーマのバリデーション | 3ツール全てで実装（共通スキーマを信頼の起点に） |

## なぜ furui-lite に AI 機能を入れないか

- 仕分け作業中に「もう少し候補を増やしたい」と感じても、それは収集ツールの仕事として独立させた方が、UI も依存もシンプルに保てる。
- PWA としてオフラインでも動かしたい。AI 呼び出しを抱えるとオフライン要件と衝突する。
- 収集と仕分けで責務が分かれていれば、どちらかの実装を入れ替えても他方に影響しない。
